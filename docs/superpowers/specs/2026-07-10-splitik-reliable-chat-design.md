# Надёжный чат Splitik: дизайн

> Исторический дизайн. PWA-контур удалён 2026-07-14; актуальный клиент — нативное iOS-приложение.

## Цель

Сделать мобильный чат Splitik предсказуемым как мессенджер: клавиатура не скрывает последние сообщения и поле ввода, iPhone не увеличивает интерфейс при наборе, а сбой сети или AI не лишает пользователя пути создать событие.

## Границы

- Работа ведётся только в backend-served Next.js PWA и FastAPI API этого репозитория.
- Визуальный язык остаётся текущим: синий фон, белый композер, карточки черновиков.
- Не отключать системный pinch-to-zoom: это важно для доступности. Устранить только автоматический Safari zoom у текстового поля.
- Реальные изменения данных по-прежнему возможны только через существующее подтверждение черновика.

## Чат

`SplitikScreen` становится самостоятельной экранной областью, высота которой синхронизируется с `window.visualViewport`. Внутри неё три зоны: шапка, прокручиваемый список сообщений и композер в обычном flex-потоке. Композер больше не фиксируется к layout viewport, поэтому поднимается вместе с видимой областью над iOS-клавиатурой.

При фокусе поля, изменении visual viewport, отправке своего сообщения и новом ответе список прокручивается к концу. Если пользователь вручную ушёл далеко вверх, новый ответ не должен насильно возвращать его вниз. Поле использует 16px на телефоне, не имеет ручного `resize`, а его высота растёт только до заданного лимита.

## Ошибки и резервный путь

Сообщение не исчезает после сетевого сбоя: UI сохраняет его как неотправленное и показывает действие «Повторить». Для ответа Splitik, который не может быть получен из-за сети, таймаута или временной ошибки провайдера, отображается ясное объяснение и ссылка «Создать событие вручную». Ссылка открывает существующий сценарий создания события и переносит текст пользователя в поле названия настолько, насколько это можно сделать безопасно.

## Короткий путь к черновику

Явная команда создать событие (например, «создай событие Такси до дома») обрабатывается сервером до LLM-planner: backend детерминированно выделяет название, создаёт pending `create_event` draft и возвращает карточку подтверждения. Неявные и сложные фразы по-прежнему может уточнять модель, но основной сценарий не проходит через цепочку intent model -> planner -> fallback model.

## Надёжность Splitik

- Использовать уже загруженную сессию при построении контекста вместо лишнего чтения, которое может вернуть ложный `Splitik session not found`.
- Для простого чата оставить быстрый LLM путь; для явного создания события убрать несколько последовательных вызовов модели.
- Сохранять диагностические различия между ошибкой сети на клиенте, ошибкой API и ошибкой LLM-provider, не показывая пользователю технические детали.

## Проверка

- Контрактные PWA-тесты: размер текста ввода, отсутствие fixed-композера, viewport-синхронизация, доступная ссылка ручного создания и повторная отправка.
- Backend-тесты: явная команда создания события возвращает pending draft без вызова LLM; построение контекста не перечитывает уже загруженную сессию.
- Ручная проверка на мобильном viewport: фокус поля, показ клавиатуры, отправка и открытие карточки черновика.
