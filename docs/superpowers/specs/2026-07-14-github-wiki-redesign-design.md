# Перестройка GitHub Wiki SplitAppBackend — проектирование

## Цель

Полностью заменить текущую структуру GitHub Wiki для `SplitAppBackend` на подробную русскоязычную базу знаний, в которой продуктовые правила объясняются прежде технических деталей, а каждый важный факт можно проверить по ссылке на исходный код, OpenAPI-контракт или связанную страницу.

Результат должен быть доступен в [GitHub Wiki SplitAppBackend](https://github.com/Strongf-bob/SplitAppBackend/wiki) и иметь зеркальный, редактируемый источник в `docs/wiki/` данного репозитория.

## Контекст и границы

`SplitAppBackend` — источник правды для авторизации, доступа, событий, чеков, балансов, платежей, споров, уведомлений и подтверждаемых действий агента Splitik. Нативное приложение находится в отдельном репозитории [Strongf-bob/SplitApp](https://github.com/Strongf-bob/SplitApp); вики должна описывать его только как клиент backend API, не подменяя его собственную документацию.

В работу входят:

- полный пересмотр всех Markdown-страниц `docs/wiki/`;
- публикация тех же страниц в GitHub Wiki через Git-репозиторий `SplitAppBackend.wiki.git`;
- русскоязычная продуктовая, интеграционная, техническая и эксплуатационная документация;
- внутренняя навигация между страницами и абсолютные ссылки на GitHub-исходники, `openapi.yaml` и связанный iOS-репозиторий;
- диаграммы Mermaid, когда они делают поток, состояния или границы ответственности понятнее.

В работу не входят изменения поведения API, iOS-кода, production-конфигурации или визуального сайта `docs/business-logic-site/`.

## Принципы содержания

| Принцип | Решение |
| --- | --- |
| Язык | Весь объясняющий текст, заголовки, таблицы и подписи — на русском. Идентификаторы, команды, URL, имена полей и HTTP-методы сохраняются как в коде. |
| Источник правды | Утверждения о работающем поведении подтверждаются ссылкой на конкретный файл/строку GitHub либо на `openapi.yaml`. Предположения явно помечаются как планы или ограничения. |
| Порядок изложения | Сначала смысл для пользователя и бизнес-правила, затем граница клиента и сервера, и только после этого реализация и эксплуатация. |
| Ссылки | Внутренние Wiki-ссылки используют имена страниц без `.md`; внешние ссылки на файлы репозитория сохраняют `.md` и ведут на `main`. |
| Безопасность | Вики не содержит секретов, реальных токенов, персональных данных, дампов БД и небезопасных примеров обхода контроля доступа. |
| Поддерживаемость | `docs/wiki/` — единственный редактируемый источник. GitHub Wiki — опубликованное зеркало. |

## Навигационные и ссылочные правила

1. `Home.md` — единственная точка первого входа; она ссылается на все верхнеуровневые страницы.
2. Каждая страница заканчивается таблицей «Связанные страницы» с обратными ссылками на ближайшие темы.
3. Ссылка на файл backend имеет вид `https://github.com/Strongf-bob/SplitAppBackend/blob/main/<path>#L<line>`; на iOS-код — аналогичный URL в `Strongf-bob/SplitApp`.
4. API-детали не дублируют весь OpenAPI: страница объясняет назначение и правила, а полный контракт ведёт на [`openapi.yaml`](https://github.com/Strongf-bob/SplitAppBackend/blob/main/openapi.yaml).
5. Все внешние Markdown-ссылки проходят проверку на существование целевого файла; внутренние — на наличие целевой Wiki-страницы.

## Требования к глубине и визуализации

- Каждая продуктовая и техническая страница имеет краткое резюме, таблицы, ссылки на первоисточники и раздел «Связанные страницы».
- Диаграммы отображают только факты, подтверждённые исходниками: пользовательский путь — `flowchart`/`journey`, запрос — `sequenceDiagram` с `autonumber`, модель — `erDiagram`, статусы — `stateDiagram-v2`.
- Темная палитра Mermaid использует фон `#2d333b`, границу `#6d5dfc`, текст `#e6edf3`, фон подграфов `#161b22` и линии `#8b949e`.
- Для каждой диаграммы добавляется HTML-комментарий с исходниками; для таблиц компонентов и API — столбец «Источник».
- Executive и Product Manager страницы обходятся без фрагментов кода; Developer и Staff страницы включают только короткие объясняющие примеры, необходимые для работы.

## Технический процесс публикации

1. Переписать Markdown-источник в `docs/wiki/` согласно архитектуре выше.
2. Проверить: frontmatter, русскоязычные заголовки, внутренние ссылки, внешние GitHub URL, существование упомянутых файлов и синтаксис Mermaid.
3. Закоммитить только отслеживаемые исходники документации в основном репозитории.
4. Клонировать GitHub Wiki, зеркально заменить Markdown-страницы, закоммитить и отправить изменения в `https://github.com/Strongf-bob/SplitAppBackend.wiki.git`.
5. Убедиться через `git ls-remote` и публичные URL, что опубликована ожидаемая версия.

## Критерии готовности

- GitHub Wiki открывается с новой русскоязычной `Home` и навигацией по шести разделам.
- Все перечисленные страницы существуют и содержат рабочие внутренние ссылки.
- `SplitApp-Integration` прямо объясняет связь `SplitAppBackend` и `SplitApp`, их ответственность и правила контракта.
- Бизнес-логика расходов, денег, платежей, споров, закрытия и Splitik изложена отдельно от API-реализации.
- Технические страницы ссылаются на действующие файлы backend и полный OpenAPI-контракт.
- Локальный источник и опубликованная GitHub Wiki совпадают по Markdown-страницам.
