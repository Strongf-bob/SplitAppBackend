Auth, access, invariants

Кто имеет право менять деньги и участников

Backend не доверяет user id из payload как доказательству прав. Каждая защищенная операция начинается с authenticated actor, затем проверяет membership, роль, статус friendship/request/receipt/payment и состояние события.

Базовая матрица прав

Операция Кто может Ключевые проверки
Создать событие Authenticated user Пользователь существует, name не пустой, политики валидны.
Читать событие, чеки, платежи, балансы Active event member assert_event_access по actor и active membership.
Менять политики или закрывать событие Creator assert_event_creator.
Добавлять участников или invites Зависит от participants_invite_policy Creator-only по умолчанию; open event; preview/accept/decline/revoke для invite lifecycle.
Создать чек Creator или participants, согласно policy Open event, payer/share users внутри event, сумма items совпадает с total.
Подтвердить чек Creator или payer, согласно policy Receipt status должен быть draft или ready_for_review.
Validate receipt и share review Authorized event member Draft переводится в pending_confirmation; каждый участник принимает или спорит только свою долю.
Создать payment Sender sender_id == actor_user_id, обе стороны участники event, sender != receiver.
Подтвердить или отклонить payment Receiver actor_user_id == receiver_id, event open, confirmed payment нельзя reject/delete.
Создать payment request Creditor creditor_id == actor_user_id, debtor и creditor разные active members события.
Acknowledge / mark-paid / extension Debtor Debtor управляет своей реакцией на request, но mark-paid создает только pending payment.
Cancel payment request Creditor Creditor может отменить request, пока lifecycle допускает cancel.
Dispute Сторона ресурса или event member Споры создаются только в рамках доступного события и ресурса.
Friend request lifecycle Requester / addressee Accept/reject делает addressee, remove/block доступны участникам friendship по сервисным правилам.

Инварианты безопасности

Actor-first authorization

Payload может содержать sender_id, payer_id, debtor_id, но эти поля всегда сверяются с actor или membership.

Closed event lock

После закрытия события финансовые mutations отклоняются, а чтение остается доступным участникам.

Soft deletes

События, чеки и платежи помечаются deleted_at, чтобы сохранить след и исключить их из active query.

Idempotency

Create receipt/payment/payment request/mark-paid используют Idempotency-Key для защиты от повторных POST.

Confirmation summaries

Опасные confirm/void/reject действия имеют отдельный summary endpoint перед mutation.

Visibility-limited users

Список users не является dump всей базы: видны actor и пользователи из общих событий, search работает opt-in.

Валидация данных

Чеки

Позиции обязательны, стоимость больше 0, сумма всех positions должна совпадать с total, share values каждой позиции должны суммироваться в 1.

Пользователи

Публичный handle нормализуется, проверяется регуляркой и уникальностью. Search opt-in через discovery_enabled.

Payment requests

Creditor должен быть actor, debtor и creditor должны быть разными участниками event, deadline с timezone и минимум 30 минут вперед.

Receipt images

Изображения чеков хранятся приватно. Чтение идет через presigned URL, replacement/delete должны убирать старое состояние storage.

Контакты

Импорт контактов owner-scoped: сохраняется display name, нормализованный phone/hash и optional matched user, но не глобальная адресная книга.

Splitik

Агент получает bounded context после membership checks; commit draft снова проходит обычную backend-доменную логику.

Операционная надежность

Механизм Зачем нужен
Idempotency key Повторный POST после сетевого сбоя не должен создать второй чек, payment или payment request.
Audit events Критичные действия вроде delete, confirm, profile update оставляют серверный след.
Metrics Backend пишет service/domain metrics, request metrics и money observations.
Generic unexpected errors Клиент не получает внутренние детали неожиданных failure, но сервер логирует request context.
