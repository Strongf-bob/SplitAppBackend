Ledger и balances

Как backend превращает чеки и платежи в долги

Денежная модель разделяет confirmed, pending и disputed. Confirmed receipt создает долг участников перед payer, confirmed payment создает встречный перенос от receiver к sender, а pending/disputed показываются отдельно и не смешиваются с итоговым долгом.

Формула на уровне позиции чека

Позиция: пицца 1 200 руб. Платит Alice. Доли: Alice 25%, Bob 75%.

Alice -> Alice: 300 руб.  игнорируется, потому что debtor == creditor
Bob   -> Alice: 900 руб.  попадает в ledger как долг Bob перед Alice

Backend хранит деньги в копейках и округляет доли до целых копеек через Decimal с ROUND_HALF_UP.

Home summary: confirmed / pending / disputed

GET /api/home/summary возвращает три money bucket вместо одной общей суммы:

{
  "confirmed": { "owed_kopecks": 70000, "receivable_kopecks": 120000 },
  "pending": { "owed_kopecks": 25000, "receivable_kopecks": 0 },
  "disputed": { "owed_kopecks": 0, "receivable_kopecks": 40000 }
}

Это нужно, чтобы UI не показывал неподтвержденный или спорный долг как финальный.

Алгоритм balances

  1. Загрузить source documents. Backend берет active confirmed receipts и active confirmed payments события.
  2. Построить ledger. Для каждой доли позиции добавляется transfer debtor -> payer.
  3. Учесть payments. Подтвержденный payment добавляет transfer receiver -> sender, то есть обратное направление долга.
  4. Схлопнуть пары. Если Bob должен Alice 900, а Alice должна Bob 200, итог: Bob должен Alice 700.
  5. Вернуть строки. API возвращает event_id, debitor_id, creditor_id, amount_kopecks.

Payment request flow

Request

Creditor создает запрос к debtor. Creditor должен совпадать с authenticated actor.

Acknowledge

Debtor может отметить, что запрос увиден. Это не меняет баланс.

Mark paid

Debtor создает pending payment, связанный с payment request.

Confirm

Receiver подтверждает payment, и только тогда он влияет на balances.

Void и correction

Void confirmed receipt

Creator или payer может void confirmed receipt. После этого receipt выходит из confirmed balances, а audit фиксирует действие.

Correction draft

Correction создает новый draft на основе confirmed receipt и помечает исходный receipt как corrected. Новый вариант снова проходит review/confirm.

Confirmation summary

Перед confirm/void receipt и confirm/reject payment backend отдает summary с warnings, amount и requires_explicit_confirmation.

Balance explain

/api/events/{id}/balances/explain возвращает contributions, чтобы показать, из каких receipt/payment строк собран долг.
