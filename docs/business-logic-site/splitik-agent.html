ИИ-агент SplitApp

Сплитик уже работает как контекстный агент с безопасными draft actions

Он читает только разрешенный backend-контекст, помогает с событиями, чеками и долгами, создает черновики действий и не меняет реальные деньги без явного подтверждения.

Splitik runtime backend-owned
1 Пользователь пишет

Текст, режим и entry point: общий экран, событие, чек или участник.

2 Backend собирает факты

Actor, membership, последние сообщения, active draft, balances и tool results.

3 Ответ плюс draft

Модель формулирует ответ, а backend хранит и подтверждает только валидные действия.

Backend policy

Проверяет доступ, режим, session и допустимые capabilities.

→
LLM adapter

Получает только bounded context и системные правила Сплитика.

→
Draft boundary

Создает или обновляет pending draft. Деньги не меняются.

Что Сплитик умеет сейчас

? Объяснять события и долги

Отвечает на вопросы вроде “кто мне должен” по backend-computed balances, а не по догадкам модели.

+ Создавать draft события

В general mode может подготовить create_event draft из текста пользователя.

₽ Готовить draft чека

В event mode создает create_receipt draft из текста или image attachment.

! Блокировать опасные ответы

Refusal для домашки, секретов, приватных трат и заявлений модели “я изменил баланс”.

Режимы контекста

general Профиль и общий обзор

Текущий пользователь, его события и accepted friends.

event Событие

Участники, receipts, balances, explanations и история события после membership check.

receipt Чек

Детали расхода, позиции, share items и связанное событие.

member Участник

Контекст конкретного участника только внутри общего видимого события.

A → B → draft: реальные сценарии

1. Создать событие из фразы

Сплитик не создает событие сразу. Он готовит pending draft, который нужно подтвердить.

general mode
A: ввели 
{
  "mode": "general",
  "message": "Создай событие: Ужин в Duo",
  "locale": "ru-RU",
  "timezone": "Europe/Moscow"
}
B: backend ответил 
{
  "intent": "draft",
  "assistant_message": "Подготовил черновик события.",
  "guardrail_decision": {
    "allowed": true,
    "reason": "allowed"
  }
}
Draft 
{
  "type": "create_event",
  "status": "pending",
  "payload": {
    "name": "Ужин в Duo"
  },
  "version": 1,
  "source": "text"
}

6. Спросить, кто мне должен

Расчеты делает backend. LLM только переводит уже посчитанные balances в понятный ответ.

explain
A: спросили 
{
  "mode": "general",
  "message": "Кто мне должен деньги?"
}
B: backend context 
{
  "user_balance_summary": {
    "outstanding_owed_kopecks": 0,
    "outstanding_receivable_kopecks": 5000,
    "events": [
      {
        "event": { "name": "Trip" },
        "balances": [
          {
            "debitor_id": "user-b",
            "creditor_id": "user-a",
            "amount_kopecks": 5000
          }
        ]
      }
    ]
  }
}
Ответ 
Сплитик объясняет:
"В Trip вам должны 50 рублей.
Расчет взят из подтвержденных чеков
и подтвержденных платежей."

Граница commit: где реально меняется состояние

Обычный ответ Сплитика не создает деньги, не подтверждает чек и не закрывает долг. State-changing операция проходит через отдельный backend endpoint.

До commit

В базе есть splitik_drafts со статусом pending. Пользователь может проверить и отредактировать payload.

→
POST /api/splitik/drafts/{id}/commit

Backend проверяет owner, pending status и тип draft, затем создает событие или receipt draft в обычной доменной логике.

Что Сплитик специально не может

Не удаляет события по просьбе модели

Запрещенные capabilities включают forbidden:delete_event и прямое изменение money state.

Не раскрывает чужие приватные траты

Вне общего события запросы про личные траты другого пользователя блокируются.

Не верит клиентскому user id

Actor берется из auth dependency, а доступ к событию, чеку и draft проверяется на backend.

Пример refusal

Если запрос не про SplitApp, backend отказывает до LLM-вызова.
